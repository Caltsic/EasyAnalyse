# AI原生电路交换格式设计说明

文档版本：`v4.0.0`  
文档状态：现行设计说明  
规范入口：[exchange.md](./exchange.md)  
正式 Schema：[AI原生电路交换格式.schema.json](./AI原生电路交换格式.schema.json)

## 1. 文档定位

这份文档解释 EASYAnalyse 当前交换格式为什么这样设计，以及它服务的产品目标是什么。

当前真正有约束力的来源只有三处：

- `exchange.md`
- 正式 schema
- 运行时的归一化与校验实现

本文不再单独定义第二套字段，也不覆盖运行时行为。

## 2. 为什么坚持语义优先

这个项目不是为了复刻传统 EDA 的导线拓扑编辑，而是为了让 AI 与工程师围绕“器件关系”而不是“布线几何”协作。

如果把导线、节点、折点、独立 signal 对象当成主模型，会立刻出现几个问题：

1. AI 必须先伪造几何结构，再表达真实电路语义，出错概率高。
2. 工程师读图时会被导线噪声拖慢，而不是直接看到模块关系。
3. 保存格式、样例、编辑器行为很容易演化成多套并行模型。

因此，现行模型只保留以下核心对象：

- `document`
- `devices`
- `terminals`
- `view`

其中：

- `devices` 表达器件、模块、芯片、连接器和基础元件
- `terminals` 表达每个器件的接口
- `label` 是连接语义本身
- `view` 只表达阅读层，而不表达电气真值

## 3. 当前模型的关键取舍

### 3.1 连通关系只看 terminal label

如果两个 terminal 共享同一个非空 `label`，它们就被视为连通。

这意味着：

- 不再持久化 `signals`
- 不再持久化 `signalId`
- 不再依赖 wires、junctions、折点来定义电路真值

### 3.2 电路必须带有可判断的具体值

对 EASYAnalyse 而言，一个“有名字但没数值”的器件并不能构成有意义的电路描述。

当前设计要求至少做到：

- 电阻、电容、电感等基础无源器件写出 `properties.value`
- 晶振、振荡器、时钟器件写出 `properties.frequency`
- 电源和稳压相关器件写出 `properties.voltage`、`properties.outputVoltage` 或 `properties.nominalVoltage`
- 如果网络标签写的是泛化名称如 `VCC`、`VIN`，也必须从供电器件属性中补出具体电压

换句话说，格式不是只为了“画个框图”，而是为了表达能被人判断功能含义的电路。

### 3.3 方向模型拆分为物理方向和可选阅读流向

现行模型保留 terminal 的物理方向 `direction`，并允许在需要时增加可选阅读流向 `logicalDirection`。

`direction` 允许：

- `input`
- `output`
- `bidirectional`
- `passive`
- `power-in`
- `power-out`
- `ground`
- `shield`
- `unspecified`

其中：

- I2C `SCL`、`SDA` 这类总线端子优先写成 `bidirectional`
- 电阻、电容、电感、磁珠、晶振、开关等两端无源器件优先写成 `passive`
- 电源输入可写 `power-in`
- 电源输出可写 `power-out`
- 地端可写 `ground`

`logicalDirection` 只在“物理端子本身还是无源或双向，但聚焦阅读需要稳定左右语义”时才补充。

例如：

- 串联电阻、反馈电阻这类需要清楚表达上游和下游关系的器件，可以保留 `direction = passive`，再按阅读方向补 `logicalDirection`
- 纯去耦、电源旁路、对地泄放这类没有稳定左右语义的器件，可以只保留 `passive`，不强行补逻辑方向

布局默认侧边不再只看 `input / output`，而是优先看可折叠成源/汇角色的 `logicalDirection`，否则再看 `direction`。
聚焦时只保留左列和右列：同向同级器件不出现，同时跨上下游两侧的器件优先归到右侧。

### 3.4 视图层不保存自由端子坐标

当前编辑器允许用户调整端子在器件边缘上的相对位置，但格式层不持久化任意 `x/y` 端子坐标。

持久化字段只保留：

- `side`
- `order`

这背后的原因是：

1. 端子必须绑定在器件边缘，不能脱离器件自由漂移。
2. 同一个文档在不同窗口尺寸与排版策略下，最终标签避让与引线布局本来就是派生结果。
3. 如果把端子与标签的像素级坐标固化进格式，会让生成与维护复杂度急剧上升。

因此：

- 端子标签应该自动跟随端子
- 标签避让、引线、层级和防重叠属于渲染派生行为
- 这些结果不应成为交换格式主数据

## 4. `view` 层的职责

`view` 是阅读层，不是隐藏的电路真值层。

它负责：

- 器件位置与尺寸
- 显示旋转角度与形状偏好
- 画布与网格设置
- 独立网络线 `view.networkLines`

它不负责：

- 导线拓扑
- 节点图
- 网表替代
- PCB 走线约束
- 仿真级连线几何

`view.networkLines` 的意义也必须明确：

- 它是线性的独立公共网络对象
- 它与器件同级存在
- 它通过 `label` 指向共享网络语义
- 它帮助聚焦与阅读，但不改变连通真值

推荐优先用 `3V3`、`5V`、`12V` 这样的具体标签，而不是把文档主示例写成没有电压上下文的 `VCC`。

## 5. 保存与校验策略

保存不是 advisory 模式。当前实现会先归一化，再检查 schema 与语义约束，只有两者都通过才允许写盘。

实际保存门槛是：

- `schemaValid = true`
- `semanticValid = true`

当前后端里，`semanticValid` 是通过“是否还存在任何 `semantic.*` 问题”计算出来的，因此当前语义问题必须清零。

现行语义检查重点包括：

1. ID 是否重复
2. `view.devices` 是否引用了不存在的 device
3. 聚焦目标是否引用了不存在的 device
4. 必填端子是否缺少标签
5. 无源器件是否缺少具体值
6. 时钟器件是否缺少具体频率
7. 泛化电源标签是否能解析到具体电压
8. 独立网络线是否为空标签或未被任何 terminal 使用

因此，格式文档、示例文档和编辑器默认行为都必须朝“可通过保存门槛”这个方向收敛，而不是制造一批只能显示、不能保存的样例。

## 6. 面向 AI 的要求

如果 AI 要直接生成 EASYAnalyse 文档，至少要满足下面几点：

1. 只输出现行 `4.0.0` 结构。
2. 不输出旧的 `components / ports / nodes / wires / signals / signalId`。
3. 电阻、电容、电感、晶振、电源等关键器件带出明确参数。
4. 端子先表达物理角色：无源器件用 `passive`，双向总线用 `bidirectional`；只有在阅读或聚焦需要明确上下游时，才补 `logicalDirection`。
5. 端子标签直接使用共享网络语义。
6. 用 `view` 提供可阅读的布局，而不是伪造导线。

## 7. 这套格式不承担什么

现行交换格式刻意不承担以下职责：

- 传统原理图导线绘制系统
- 完整仿真网表
- PCB 布线约束
- 完整 EDA 元件库语义

它的目标不是替代所有硬件工具，而是在 AI 与工程师之间建立一个足够真实、足够可读、足够可校验的中间表达。

## 8. 结论

现行 `semantic v4` 的核心价值不在于“把线画得像原理图”，而在于：

- 用 terminal label 统一连接语义
- 用具体值保证电路有判断意义
- 用 `direction` 表达物理端子角色，必要时用 `logicalDirection` 表达阅读流向
- 用 `view` 承担阅读性而不是电气真值

只有把这些边界收紧，AI 生成的结果和工程师实际需要的电路表达才会收敛到同一套数据模型里。
